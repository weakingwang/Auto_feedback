# 调试与发布习惯总结（Auto Feedback）

这份文档沉淀本项目收尾阶段验证过的高效做法，用于后续项目的总体方向参考。

## 1. 先本地闭环，再上服务器

- 先在本地完成完整链路验证：前端、后端、上传、预览、列表回显。
- 本地闭环后再部署，能把问题范围从“代码+环境”快速收敛到“部署环境/发布流程”。

## 2. 路径问题是一等公民

- 上传链路最常见故障来自路径不一致：运行目录、挂载目录、代码拼接路径。
- 实践建议：读取上传文件时优先使用 `multer` 返回的 `file.path`，避免手工拼接导致 `ENOENT`。
- 部署时保证 `UPLOAD_DIR`、容器挂载路径、静态资源映射三者一致。

## 3. 发布要“重建 + 重启”，不要只重启

- 代码改动后必须执行镜像重建，再启动容器。
- 推荐流程：
  - `docker compose build backend frontend`
  - `docker compose up -d mysql backend frontend`
- 仅 `restart` 容器通常不会加载新代码。

## 4. 用日志做验收，而不只看页面

- 页面“看起来正常”不等于链路无错误。
- 验收时同步观察后端日志，确认关键错误是否消失（如 `ENOENT /app/uploads/...`）。
- 常用命令：
  - `docker compose logs -f --tail=200 backend frontend`

## 5. 容器冲突先清理旧状态

- 遇到 `container name already in use`，说明旧容器残留。
- 标准处理：
  - `docker compose down`
  - 必要时手动清理同名容器
  - 再执行 build/up
- 避免在脏状态下反复尝试启动。

## 6. 小改动，小步快验

- UI 诉求（例如左右栏不等高）优先做最小改动（CSS 层）。
- 每次改完立即刷新验证，确认“改动范围最小、效果可见、无副作用”。

## 7. 推荐最小发布验收清单（3-5 分钟）

1. 打开首页，确认前端可访问。
2. 提交一条带图片反馈。
3. 确认反馈区出现缩略图并可点开原图。
4. 查看后端日志确认无上传/路径错误。
5. 访问 `health` 接口确认服务稳定。

---

如果后续项目也遵循这套顺序（本地闭环 -> 重建发布 -> 日志验收），定位效率和交付稳定性会明显提升。
